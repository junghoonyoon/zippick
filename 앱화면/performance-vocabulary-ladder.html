Vocabulary Ladder

“사이트가 느려요”에서 “무엇을 측정하고 바꿀까요?”까지

개발에 익숙하지 않아도 성능 대화는 잘할 수 있어요. 핵심은 정답 용어를 외우는 게 아니라, 막연한 느낌을 관찰, 지표, 가설, 실험으로 한 칸씩 올리는 것입니다.

Unknown Unknowns

모르는 것을 줄이는 가장 빠른 방법은 “분류”입니다

속도 문제는 보통 한 가지 원인이 아닙니다. 네트워크, 이미지, 자바스크립트, 서버 응답, 화면 그리기, 체감 흐름이 겹쳐서 나타납니다.

처음에는 이렇게만 나눠도 충분합니다

  • 처음 화면이 늦게 보이는 문제
  • 버튼이나 검색 입력 반응이 늦는 문제
  • 화면이 흔들리거나 갑자기 밀리는 문제
  • 데이터를 기다리느라 비어 보이는 문제
  • 내 폰에서는 괜찮은데 실제 사용자에게 느린 문제
The Ladder

좋은 프롬프트는 사다리를 올라갑니다

“빨라지게 해줘”도 시작점으로는 괜찮습니다. 다만 다음 단계의 말이 붙을수록 내가 원하는 도움의 정확도가 크게 올라갑니다.

0 느낌모바일에서 느려 보여.
1 상황검색 결과 화면 첫 진입이 느려.
2 지표LCP, INP, CLS 중 무엇이 나쁜지 확인해줘.
3 원인 후보이미지, API, JS 실행, 렌더링 중 병목을 나눠줘.
4 제약디자인은 유지하고 모바일 4G 기준으로 개선하고 싶어.
5 실험가장 영향 큰 3개를 고르고 측정 전후를 비교해줘.
Core Web Vitals

성능 대화의 기본 좌표 3개

LCP 첫 핵심 화면 사용자가 “이제 보인다”고 느끼는 큰 콘텐츠가 뜨는 시간입니다. 히어로, 대표 이미지, 첫 결과 카드가 자주 원인입니다.
INP 반응성 터치, 입력, 클릭 뒤 화면이 반응하기까지의 지연입니다. 무거운 스크립트나 렌더링 작업이 주범일 수 있습니다.
CLS 흔들림 이미 보인 화면이 갑자기 밀리는 정도입니다. 이미지 크기 미지정, 늦게 뜨는 배너, 폰트 교체가 흔한 원인입니다.
프롬프트: 이 모바일 페이지를 Core Web Vitals 관점에서 봐줘. LCP, INP, CLS 각각에 대해 가능성 높은 원인과 확인 방법, 수정 우선순위를 알려줘.
Waterfall

“느림”을 시간표로 바꾸면 대화가 쉬워집니다

네트워크 워터폴은 브라우저가 어떤 파일을 언제 가져왔는지 보여주는 시간표입니다. 여기서 큰 파일, 늦게 시작한 요청, 줄줄이 기다리는 요청을 찾습니다.

Mobile Reality

모바일 성능은 “작은 화면” 문제가 아닙니다

폰은 네트워크가 흔들리고, CPU가 약하고, 배터리 절약 모드가 있고, 스크롤과 터치를 계속 처리해야 합니다. 데스크톱에서 빠른 사이트도 모바일에서는 느릴 수 있습니다.

Common Bottlenecks

모바일 사이트에서 자주 숨어 있는 병목

이미지

너무 큼

실제 표시 크기보다 큰 이미지를 받고 있거나, 최신 포맷과 지연 로딩을 쓰지 않을 수 있습니다.

자바스크립트

너무 오래 실행

처음 화면에 필요 없는 코드까지 내려받고 실행하면 입력 반응이 둔해집니다.

API

기다림이 김

검색, 필터, 상세 데이터 요청이 직렬로 이어지면 화면이 빈 채로 오래 보입니다.

캐시

매번 새로 받음

잘 안 바뀌는 데이터와 파일도 매번 받아오면 체감 속도가 흔들립니다.

폰트

글자가 늦게 뜸

웹폰트가 늦으면 빈 글자, 글자 교체, 레이아웃 이동이 생길 수 있습니다.

체감 흐름

기다림이 불안함

스켈레톤, 즉시 피드백, 이전 결과 유지가 없으면 같은 시간도 더 느리게 느껴집니다.

Prompt Upgrade

프롬프트를 이렇게 올려 말해보세요

낮은 사다리

모바일에서 느린데 최적화해줘.

가능은 하지만 범위가 너무 넓어서, AI가 임의로 문제를 정하거나 눈에 보이는 코드만 만지기 쉽습니다.

높은 사다리

모바일 검색 결과 첫 진입이 느려 보여. LCP/INP/CLS와 네트워크 요청 관점에서 병목 후보를 먼저 진단하고, 디자인을 유지한 채 영향 큰 순서로 수정해줘. 수정 후 측정 방법도 같이 남겨줘.

증상, 화면, 지표, 제약, 검증 방법이 들어 있어 결과가 훨씬 좋아집니다.

Measurement

속도 개선 요청에는 “측정해줘”가 들어가야 합니다

확인할 것

  • 실제 모바일 크기에서 첫 화면이 언제 보이는지
  • 느린 네트워크 조건에서도 쓸 만한지
  • 초기 파일 크기와 이미지 크기가 과한지
  • 검색, 필터, 상세 보기 같은 핵심 흐름이 버벅이는지
  • 수정 전후에 좋아졌다는 근거가 있는지
관찰 → 측정 → 원인 후보 → 작은 수정 → 다시 측정 성능 개선은 한 번에 “감”으로 끝내는 작업보다, 작게 가정하고 확인하는 작업에 가깝습니다.
Ready-To-Use Prompts

바로 붙여 넣기 좋은 프롬프트 4개

진단: 이 모바일 페이지의 속도 병목을 먼저 진단해줘. 첫 화면, 입력 반응, 화면 흔들림, API 대기, 이미지 크기 관점으로 나눠서 가능성 높은 순서로 설명해줘.
우선순위: 내가 개발에 익숙하지 않으니, 수정 난이도와 효과를 함께 평가해서 오늘 할 일 3개만 골라줘. 위험한 변경은 따로 표시해줘.
측정: 수정 전후를 비교할 수 있게 어떤 지표를 보면 되는지 알려줘. 모바일 기준으로 확인 순서와 기대되는 개선 신호를 써줘.
제약: 현재 디자인과 사용자 흐름은 유지하고 싶어. 눈에 보이는 변화가 적으면서 체감 속도에 영향 큰 개선부터 제안하고 적용해줘.
Your Operating System

앞으로 이렇게 요청하면 됩니다

나는 개발에 익숙하지 않고, 모바일 사용자가 빠르게 느끼는 것이 중요해. 이 화면에서 속도 문제의 unknown unknown을 찾아줘. 먼저 증상을 분류하고, Core Web Vitals와 네트워크/이미지/JS/API/렌더링 관점으로 병목 후보를 세운 뒤, 효과가 큰 순서로 작은 수정안을 적용해줘. 마지막에는 내가 이해할 수 있는 말로 “무엇이 느렸고, 무엇을 바꿨고, 어떻게 확인하면 되는지”를 남겨줘.

이 문장 하나면 “빨리 해줘”보다 훨씬 좋은 출발점이 됩니다. 이제 성능 대화의 목적은 용어를 멋지게 쓰는 게 아니라, AI가 문제를 제대로 찾도록 질문의 모양을 잡는 것입니다.
